# excessive-validator-params

**Severity:** Info | **CWE:** -

## What it detects

Validators with more than 4 deployment-time parameters. The threshold of 4 is the recommended maximum for maintaining manageable validator complexity.

## Why it matters

Every validator parameter in Aiken is baked into the compiled script at deployment time. Having too many parameters creates several practical problems:

- **Deployment complexity**: Each parameter must be supplied correctly during deployment. More parameters means more opportunities for misconfiguration, especially when parameters have similar types (e.g., multiple `ByteArray` key hashes).
- **Increased script size**: Each parameter adds to the UPLC script size, which directly increases the cost of every transaction that interacts with the contract. For frequently used validators, this adds up.
- **Address fragmentation**: Since parameters are part of the script hash, any parameter change produces a different script address. With many parameters, testing and deployment across environments becomes harder to manage.
- **Design smell**: A validator with many parameters often indicates that configuration should be moved to a datum, a reference UTXO, or a parameterized reference script instead of being hard-coded at deployment.

## Example: Flagged Code

```aiken
validator complex_dex(
  admin_pkh: ByteArray,
  treasury_pkh: ByteArray,
  oracle_policy: ByteArray,
  lp_token_policy: ByteArray,
  fee_address: Address,
) {
  spend(datum: Option<DexDatum>, redeemer: DexRedeemer, own_ref: OutputReference, self: Transaction) {
    expect Some(d) = datum
    // All 5 parameters used across various branches...
    when redeemer is {
      Swap -> verify_swap(d, self, oracle_policy, fee_address)
      AddLiquidity -> verify_add_liq(d, self, lp_token_policy, treasury_pkh)
      RemoveLiquidity -> verify_remove_liq(d, self, lp_token_policy, admin_pkh)
      UpdateConfig -> list.has(self.extra_signatories, admin_pkh)
    }
  }
}
```

## Example: Improved Code

```aiken
// Group related parameters into a configuration type stored in the datum
type DexConfig {
  admin_pkh: ByteArray,
  treasury_pkh: ByteArray,
  oracle_policy: ByteArray,
  lp_token_policy: ByteArray,
  fee_address: Address,
}

type DexDatum {
  config: DexConfig,
  pool_state: PoolState,
}

// Only the essential parameter (or none) at the validator level
validator simple_dex {
  spend(datum: Option<DexDatum>, redeemer: DexRedeemer, own_ref: OutputReference, self: Transaction) {
    expect Some(d) = datum
    when redeemer is {
      Swap -> verify_swap(d, self)
      AddLiquidity -> verify_add_liq(d, self)
      RemoveLiquidity -> verify_remove_liq(d, self)
      UpdateConfig -> {
        list.has(self.extra_signatories, d.config.admin_pkh) &&
        verify_config_update(d, self)
      }
    }
  }
}
```

Alternatively, use a reference UTXO for configuration:

```aiken
// Store configuration in a reference UTXO that the validator reads at runtime
validator configurable_dex(config_token_policy: ByteArray) {
  spend(datum: Option<DexDatum>, redeemer: DexRedeemer, own_ref: OutputReference, self: Transaction) {
    expect Some(d) = datum
    // Read config from a reference input holding the config NFT
    let config_input =
      list.find(self.reference_inputs, fn(input) {
        value.quantity_of(input.output.value, config_token_policy, "config") > 0
      })
    expect InlineDatum(config_data) = config_input.output.datum
    expect config: DexConfig = config_data
    // Now use config.admin_pkh, config.oracle_policy, etc.
    ...
  }
}
```

## Detection Logic

Aikido identifies this pattern through the following steps:

1. **Scope**: Only examines validators in validator modules.
2. **Parameter count**: Compares the number of validator-level parameters (not handler parameters like datum/redeemer) against the threshold of 4.
3. **Threshold**: Fires when `validator.params.len() > 4`. Exactly 4 parameters is acceptable.
4. **Report**: Lists all parameter names in the finding description to help developers identify which parameters to group.
5. **Confidence**: Rated as `possible` because the threshold is a heuristic and some validators legitimately need 5+ independent parameters.

## False Positives

This detector may produce false positives when:

- All parameters are truly independent and cannot be logically grouped (e.g., they represent different protocol participants with no structural relationship).
- The validator is a multi-purpose contract where each parameter serves a distinct handler, and splitting into multiple validators would be worse.
- The parameters are auto-generated by a framework or code generation tool that intentionally uses many deployment-time parameters.

Suppress with:

```aiken
// aikido:ignore[excessive-validator-params] -- all params are independent protocol keys
```

Or adjust the threshold in `.aikido.toml` if your project has a different convention:

```toml
[detectors]
disable = ["excessive-validator-params"]
```
